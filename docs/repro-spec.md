# 汎用歌枠DB 再現仕様書

## 1. 文書の目的

本書は、Googleスプレッドシートで管理されている歌枠データを、Cloudflare R2 上の静的 JSON として公開し、静的 HTML で検索・閲覧できる索引型データベースとして再現するための仕様書です。

本書の目的は以下の通りです。

- 人間の実装担当者が要件を誤読しないようにする
- CODEX / GEMINI 等の AI エージェントに再現実装の前提を与える
- 別案件へ流用可能な汎用構成として整理する
- 運用・保守時の判断基準を固定する

---

## 2. このアプリの定義

このアプリは、**静的 JSON 優先**の歌枠索引データベースです。

基本構造は以下の通りです。

1. Google スプレッドシートで元データを管理する
2. 元データを整形して JSON を生成する
3. JSON を Cloudflare R2 に配置する
4. 静的 HTML / CSS / JavaScript が JSON を取得して描画する
5. ユーザーはブラウザ上で検索・閲覧する

本アプリは、以下を目的としません。

- 動的バックエンド主体のアプリ化
- 高機能 CMS 化
- 認証付き管理画面の提供
- 複雑なサーバーサイド検索基盤の構築

## 4. 全体構成

### 4.1 構成要素

本システムは、原則として以下の要素で構成します。

- Google スプレッドシート
- JSON 生成または同期用スクリプト
- Cloudflare R2
- 静的 HTML / CSS / JavaScript
- 必要な場合のみ Google Apps Script
- 必要な場合のみ GitHub Actions 等の自動化

### 4.2 データフロー

1. スプレッドシートにデータを入力・更新する
2. スクリプトまたは GAS により必要データを取得する
3. 取得データを公開用 JSON に整形する
4. JSON を R2 に配置する
5. フロントエンドが JSON を取得して描画する

### 4.3 基本原則

- スプレッドシートが一次情報源である
- JSON は公開・描画用の整形済みデータである
- フロントエンドは表示に専念する
- サーバー処理は必要最小限にとどめる
- 再現性と保守性を優先する

## 6. スプレッドシート要件

### 6.1 基本方針

スプレッドシートは本システムの一次情報源です。  
列定義、値形式、空欄ルール、表記揺れルールを曖昧にしないことを前提とします。

### 6.2 想定シート

案件に応じて増減してよいですが、基本的には以下を想定します。

- `songs`
- `archives`
- `meta`
- `gags`

### 6.3 `songs` シートの想定列

以下は標準例です。実案件では必要に応じて確定してください。

| 列名 | 内容 | 必須 |
|---|---|---|
| id | 一意識別子 | 推奨 |
| title | 曲名 | 必須 |
| artist | アーティスト名 | 必須 |
| artist_reading | 読み仮名等 | 任意 |
| date | 配信日 | 推奨 |
| archive_id | 配信識別子 | 推奨 |
| archive_url | 配信 URL | 任意 |
| notes | 備考 | 任意 |
| status | 公開状態等 | 任意 |

### 6.4 値のルール

実案件では以下を必ず明文化してください。

- 空欄は `""` とするか `null` とするか
- 日付形式は `YYYY-MM-DD` に統一するか
- 全角 / 半角の扱いをどうするか
- 空白の正規化をどうするか
- 記号・括弧・波ダッシュ等の正規化をどうするか
- 同一曲の表記揺れをどう扱うか
- 非公開データを出力対象に含めるか

### 6.5 禁止事項

以下は禁止とします。

- 列名の無断変更
- 同じ意味の列の重複作成
- 途中から日付形式を混在させること
- キー名や意味の場当たり的変更
- ドキュメント未更新のまま列構造を変えること

## 7. JSON 出力仕様

### 7.1 基本方針

JSON は、フロントエンドがそのまま扱える形で出力します。  
フロント側で複雑な再整形を前提にしないことを原則とします。

### 7.2 想定ファイル

- `public-data/songs.json`
- `public-data/archives.json`
- `public-data/meta.json`

必要に応じて追加してもよいですが、命名は明確にしてください。

### 7.3 `songs.json` の標準例

```json
[
  {
    "id": "song-0001",
    "title": "曲名",
    "artist": "アーティスト名",
    "artistReading": "",
    "date": "2026-04-01",
    "archiveId": "archive-001",
    "archiveUrl": "https://example.com",
    "notes": "",
    "status": "published"
  }
]
````

### 7.4 設計原則

* JSON キー名は固定する
* キーの命名規則は統一する
* フロントが期待するスキーマを明文化する
* 後方互換性を壊す変更は build-log に記録する
* ソートルールがある場合は出力側か表示側かを明示する

### 7.5 命名規則

以下のいずれかを案件単位で統一してください。

* `camelCase`
* `snake_case`

混在は禁止します。

### 7.6 実案件で確定すべき項目

* キー命名規則
* 空文字と空配列の使い分け
* 非公開データの扱い
* ソート済みで出力するかどうか
* ID の採番ルール
* `archives.json` / `meta.json` のスキーマ

## 9. フロントエンド描画仕様

### 9.2 主な責務

* 検索ボックス

## 10. 検索仕様

### 10.1 基本方針

検索仕様は曖昧さを残さず、再現可能な形で明記します。

### 10.2 検討対象

* 完全一致 / 部分一致
* 大文字 / 小文字の扱い
* 全角 / 半角の扱い
* 空白の扱い
* 記号除去の有無
* ひらがな / カタカナ / ローマ字の扱い
* AND / OR 条件の扱い

### 10.3 最低限確定すべき項目

以下は実装前に必ず決めてください。

* 曲名検索は部分一致か完全一致か
* アーティスト名検索は部分一致か完全一致か
* 日付検索は完全一致か
* 複数条件の結合は AND か OR か
* 検索対象外の列は何か
* 検索前正規化の内容は何か

### 10.4 実案件で埋めるべき項目

* 実際の正規化関数仕様
* 日本語検索時の表記揺れ対応方針

## 15. 実装ルール

### 15.1 原則

* 小さく作る
* 既存構造を崩さない
* スプレッドシート → JSON → R2 → HTML の流れを守る
* 変更は最小単位で行う
* スキーマ変更は明文化する
* 重要変更は build-log に残す
* 検証できないことを断定しない

### 15.2 禁止事項

* 目的外の大規模リファクタ
* 理由なきライブラリ追加
* スキーマ変更の無断実施
* ドキュメント未更新のままの構造変更
* 秘密情報の直書き
* 実在しない要件の追加

## 17. AI エージェントへの作業指示ルール

AI エージェントへ作業を依頼する際は、以下を前提にしてください。

* `AGENTS.md` を最優先で参照すること
* `docs/repro-spec.md` を要件定義として参照すること
* 非自明な変更では `docs/build-log.md` を更新すること
* 不要なリファクタを行わないこと
* 実値未確定項目は `[TBD]` のまま扱い、勝手に捏造しないこと

### 17.1 指示テンプレート

```text
Read AGENTS.md and docs/repro-spec.md first.

Work with the smallest reasonable change.

Current task:
- [describe the task]

Constraints:
- preserve the spreadsheet -> JSON -> R2 -> HTML model
- do not add unnecessary frameworks
- do not invent requirements
- do not hardcode secrets
- update docs/build-log.md for any non-trivial change
- if a required value is not defined yet, keep it as [TBD] instead of inventing one

Deliverables:
- [target files]
```
